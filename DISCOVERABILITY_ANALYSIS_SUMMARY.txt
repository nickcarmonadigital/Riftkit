PROBLEM STATEMENT
─────────────────
The AI Development Workflow Framework contains 228 skills, 13 agents, and 32
commands. When copied into a project, Claude Code discovers ZERO of them
automatically. Users must manually:
  1. Read README.md
  2. Search skills-index.md for keywords
  3. Navigate nested directory tree
  4. Explicitly request skills by name

Result: 80% of users don't know the framework exists.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ROOT CAUSES (6)
───────────────

1. NO FRAMEWORK CLAUDE.MD
   Problem: Example CLAUDE.md files exist (6 templates) but are project-specific.
   They contain project rules, style guides, and commands—not framework routing.

2. NO INTENT→SKILL MAPPING
   Problem: skills-index.md is reference documentation. There's no "when user
   says X → load skill Y" logic. AI must guess or ask explicitly.

3. NO AUTO-DISCOVERY MECHANISM
   Problem: Claude Code loads .agent/CLAUDE.md IF it exists, but doesn't
   recursively read .agent/skills/3-build/api_design/SKILL.md automatically.

4. SKILLS BURIED 5 LEVELS DEEP
   Problem: .agent/skills/3-build/api_design/SKILL.md requires explicit
   reference. Deep nesting prevents casual discovery.

5. NO PHASE-AWARE DISPATCH
   Problem: No logic to detect "I'm building a feature" → "You're in Phase 3.
   Try spec_build, tdd_workflow, or code_review."

6. NO CONTEXT BUDGET AWARENESS
   Problem: 228 skills × 200 lines = 46KB of potential context. No mechanism
   to load by phase or priority. Choose: load everything or nothing.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SOLUTION: 4-PART ARCHITECTURE
──────────────────────────────

PART 1: FRAMEWORK CLAUDE.MD (~300 lines)
├─ Loads automatically at session start (system constraint: .agent/ auto-loads)
├─ Phase-aware dispatch (Phases 0-7 + Alpha/Beta)
├─ Trigger keywords for each phase (e.g., "debug" → Phase 3 Build)
├─ Quick keyword lookup ("database" → database_optimization + observability)
└─ Links to phase registries (skills-by-phase.md)

PART 2: TRIGGER-INDEX.JSON (Machine-readable)
├─ 20-30 primary trigger patterns
├─ Maps: keywords → phase + primary skill + related skills + command
├─ Example: {"keywords": ["database", "slow"], "primary": "database_optimization"}
└─ AI-friendly for intent matching

PART 3: SKILLS-BY-PHASE.MD (Loadable registries)
├─ One section per phase (8 total)
├─ 10-25 skills per phase
├─ Grouped by category (core, language-specific, domain)
├─ Markdown links to SKILL.md files
└─ Browsable in single context load (~500 lines/phase)

PART 4: ROUTER AGENT (.agent/agents/router.md)
├─ Analyzes user prompt → detects phase + intent
├─ Matches against trigger-index.json
├─ Suggests primary + related skills
├─ Optional /router command
└─ Lightweight: phase detection + keyword matching only

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • Framework CLAUDE.md (300 lines)

CRITICAL SUCCESS FACTORS
────────────────────────

1. FRAMEWORK CLAUDE.MD MUST AUTO-LOAD
   Without it, nothing changes. System constraint ensures .agent/ folders
   auto-load in Claude Code, but example CLAUDE.md files are project-specific.

2. KEEP ROUTER LIGHTWEIGHT
   Phase detection + keyword matching only. Not ML, not deep NLP. Should be
   <100 lines of dispatch logic.

3. NEVER LOAD ALL 228 SKILLS
   Context budget killer. Phase-aware loading is non-negotiable. Never violate
   "one skill at a time" principle.

4. PRESERVE EXISTING COMMANDS
   Router augments, doesn't replace /plan, /code-review, /tdd, etc. Users who
   like commands should see router as optional enhancement.

5. KEEP SKILLS NESTED BY PHASE
   Organization is excellent. Router just surfaces them contextually via
   CLAUDE.md + trigger-index.json + phase registries.
