# Zero-Knowledge Applications

## WHEN TO USE

- Proving computation correctness without revealing private inputs
- Building privacy-preserving identity verification (age proof, credential check)
- Implementing private voting systems with verifiable tallies
- Designing ZK-rollups for blockchain scalability (proof of state transition)
- Writing arithmetic circuits in Circom, Noir, or SP1 (Succinct's zkVM)
- Optimizing proof generation time and proof size for production use
- Choosing between SNARKs (small proofs, trusted setup) and STARKs (no trusted setup, larger proofs)

## PROCESS

1. **Define the statement** — formalize what the prover wants to demonstrate: "I know x such that f(x) = y" where y is public and x is private. Identify public inputs, private witnesses, and constraints.
2. **Select proof system** — Groth16 (smallest proofs, trusted setup) for on-chain verification. PLONK/Halo2 (universal setup) for flexibility. STARKs (no setup, post-quantum) for transparency. SP1 zkVM for general-purpose programs.
3. **Write the circuit** — in Circom: define `template`, `signal`, and `<==` constraints. In Noir: write Rust-like code with `#[main]` entry. In SP1: write standard Rust, compile to RISC-V, prove execution.
4. **Compile and generate keys** — for Circom: `circom circuit.circom --r1cs --wasm --sym` then `snarkjs groth16 setup`. For Noir: `nargo compile` then `nargo prove`. For SP1: `cargo prove build`.
5. **Generate the proof** — compute the witness from private inputs, then run the prover. Optimize with GPU acceleration (rapidsnark for Groth16, stone-prover for STARKs, SP1 GPU prover).
6. **Verify the proof** — run the verifier with the proof and public inputs. For on-chain: deploy a Solidity verifier contract generated by snarkjs or Noir. Verify gas cost is acceptable.
7. **Optimize circuit size** — minimize constraint count: use lookup tables, custom gates (PLONK), or batched operations. Profile with `--inspect` flags. Reduce witness computation time.
8. **Implement recursion** — for rollups or proof aggregation, use recursive SNARKs (Halo2 IPA, Nova folding) to compress multiple proofs into one. SP1 supports native recursion.
9. **Audit and deploy** — review circuit for under-constrained signals (soundness bugs). Test with edge-case inputs. Deploy verifier contract and integrate proof generation into backend.

## CHECKLIST

- [ ] ZK statement formalized — public inputs and private witnesses defined
- [ ] Proof system chosen based on requirements (proof size, setup, post-quantum)
- [ ] Circuit compiles without errors and constraint count is reasonable
- [ ] Proof generation completes within acceptable time (< 30s for interactive use)
- [ ] Verification succeeds for valid proofs and rejects invalid ones
- [ ] On-chain verifier gas cost measured (< 300K gas for Groth16 on Ethereum)
- [ ] Circuit audited for under-constrained signals and soundness issues
- [ ] Recursive proof aggregation tested (if applicable)
- [ ] End-to-end flow tested: input, prove, verify, on-chain submit
